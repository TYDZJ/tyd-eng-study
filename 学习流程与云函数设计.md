# 学习流程与云函数设计（V1）

> 面向当前小程序学习页（`card -> learn -> spell`）与复习页（`learn -> spell`）的后端落地方案。  
> 目标：保证“每组最多 20 词、可断点续学、每次操作实时保存、下次返回同一组未完成数据”。

---

## 1. 目标与原则

- 学习模式：进入页面后获取待学习单词（最多 20），按阶段完成练习。
- 复习模式：进入页面后获取待复习单词（最多 20），按阶段完成练习。
- 断点续学：同一用户同一本词书同一模式下，存在未完成组时，始终返回该组。
- 实时保存：每次对/错或评分操作都上报后端，避免进度丢失。
- 幂等安全：弱网重试或重复点击不会造成重复记分。

---

## 2. 业务流程（V1）

## 2.1 学习模式

1. 进入页面，调用 `getOrCreateActiveSession(mode=learn)`  
2. 后端返回一组单词（`<=20`）及当前进度  
3. 前端按阶段练习：`card -> learn -> spell(可选)`  
4. 每次词操作调用 `submitWordProgress` 实时保存  
5. 全组完成后调用 `finishSession`（或后端自动判定完成）

## 2.2 复习模式

1. 进入页面，调用 `getOrCreateActiveSession(mode=review)`  
2. 后端返回一组待复习词（`<=20`）及当前进度  
3. 前端按阶段练习：`learn -> spell(可选)`  
4. 每次词操作调用 `submitWordProgress` 实时保存  
5. 全组完成后调用 `finishSession`

## 2.3 断点续学规则

- 若存在 `running` 状态的会话，优先返回该会话，不创建新组。
- 只有不存在 `running` 会话时，后端才按规则创建新组。
- 该机制确保“学习一部分后，下次进入仍返回同一组”。

---

## 3. 云函数清单（建议）

- `getOrCreateActiveSession`：获取或创建当前活动会话（核心入口）
- `submitWordProgress`：提交单词进度（核心写入）
- `finishSession`：结束会话并聚合进度
- `getTodayPlan`：获取今日学习/复习统计（用于展示）
- `getSessionState`（可选）：按 `session_id` 获取详情（调试与恢复）
- `abandonSession`（可选）：主动放弃会话

---

## 4. 数据模型设计

## 4.1 study_sessions（会话表）

用于保存“这一组最多 20 个词”的快照与总体进度。

建议字段：

- `_id`: `session_id`
- `openid`: 用户唯一标识
- `book_id`: 词书 ID（如 `cet4`）
- `mode`: `learn | review`
- `word_ids`: string[]（固定快照，长度 `<=20`）
- `target_count`: number
- `finished_count`: number
- `status`: `running | finished | aborted`
- `current_stage`: `card | learn | spell`（可选）
- `created_at` / `updated_at`

建议索引：

- `openid + book_id + mode + status`
- `openid + created_at`

## 4.2 study_session_word_progress（会话内单词进度）

用于记录每个词在当前会话中的阶段完成情况。

建议字段：

- `_id`: 自动
- `session_id`
- `openid`
- `book_id`
- `word_id`
- `card_done`: boolean
- `learn_done`: boolean
- `spell_done`: boolean
- `latest_rating`: `again | hard | good | easy`（可空）
- `submit_seq`: number（可选，用于顺序控制）
- `updated_at`

建议索引：

- `session_id + word_id`（唯一）
- `openid + session_id`

## 4.3 user_word_state（长期记忆状态）

沿用现有方案，用于调度下次复习时间。

关键字段：

- `openid`, `book_id`, `word_id`
- `status`
- `reps`, `ease_factor`, `interval_days`, `lapses`
- `last_review_at`, `next_review_at`

## 4.4 study_logs（行为日志）

记录每次提交行为，支持回放、分析与幂等追踪。

关键字段：

- `request_id`（幂等键，建议唯一）
- `session_id`, `word_id`, `stage`, `rating`
- `cost_ms`
- `created_at`

## 5. 接口契约（建议草案）

## 5.1 getOrCreateActiveSession

### 入参

```json
{
  "action": "getOrCreateActiveSession",
  "book_id": "cet4",
  "mode": "learn",
  "session_size": 20
}
```

### 规则

- `session_size` 最大 20（后端强制裁剪）
- 先查 `running` 会话，有则直接返回
- 无 `running` 会话时创建新会话并返回

### 出参

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "session_id": "sess_xxx",
    "book_id": "cet4",
    "mode": "learn",
    "status": "running",
    "target_count": 20,
    "words": [
      {
        "word_id": "talk",
        "word": "talk",
        "uk": "tɔːk",
        "us": "tɔk",
        "translations": [{ "type": "v", "translation": "说话；谈话" }],
        "phrases": [],
        "sentences": []
      }
    ],
    "progress_map": {
      "talk": {
        "card_done": false,
        "learn_done": false,
        "spell_done": false
      }
    }
  }
}
```

## 5.2 submitWordProgress

### 入参

```json
{
  "action": "submitWordProgress",
  "session_id": "sess_xxx",
  "book_id": "cet4",
  "word_id": "talk",
  "stage": "card",
  "rating": "good",
  "cost_ms": 2300,
  "request_id": "uuid-xxxx"
}
```

### 规则

- `request_id` 必传，用于幂等去重
- `stage`：`card | learn | spell`
- `rating`：`again | hard | good | easy`（阶段允许为空时需明确约束）
- 每次提交都更新会话内进度与日志
- 在 `learn/review` 阶段触发 `user_word_state` 调度更新（SM2）

```json
      "finished_count": 6,
      "target_count": 20,
      "session_finished": false
    },
    "next_review_at": "2026-04-23T10:00:00.000Z"
  }
}
```

## 6. 阶段与状态机约束（必须）

- 学习模式允许阶段顺序：`card -> learn -> spell`
- 复习模式允许阶段顺序：`learn -> spell`
- 不允许跨阶段乱提交流程（例如未 `card_done` 就写 `learn_done`，除复习模式）
- `spell` 若定义为可选，缺失时会话仍可完成（需在业务上明确）

---

## 7. 幂等与并发策略（重点）

## 7.1 幂等

- 前端每次点击生成 `request_id`（UUID）
- 后端写入前先查 `study_logs.request_id`
- 已存在则直接返回上一次结果，不重复扣分或推进进度

## 7.2 并发

- 同一用户+词书+模式仅允许 1 个 `running` 会话
- 通过事务或“先查后写 + 唯一约束”避免并发创建多个活动会话

## 7.3 断网重试

- 前端可重发同 `request_id` 请求
- 后端幂等确保结果一致

---

## 8. 与前端页面对接建议

- `index.vue`：负责调用 `getOrCreateActiveSession`，并向 `card/learn/spell` 分发 `words + progress_map + session_id`
- `card.vue`：每次对/错调用 `submitWordProgress(stage=card, rating=...)`
- `learn.vue`：每次对/错调用 `submitWordProgress(stage=learn, rating=...)`
- `spell.vue`：
  - 若坚持“纯前端”，可不提交，但会损失调度精度
  - 推荐至少在单词完成时提交 `submitWordProgress(stage=spell, rating=...)`

---

## 9. 错误码建议

- `0`：成功
- `40001`：参数错误
- `40101`：未登录/鉴权失败
- `40401`：会话不存在
- `40901`：会话状态冲突（已完成、已关闭、阶段非法）
- `90001`：服务内部异常

---

## 10. 开发顺序（推荐）

1. 新增/确认集合与索引：`study_sessions`、`study_session_word_progress`、`study_logs`
2. 先实现 `getOrCreateActiveSession`（保证断点续学）
3. 实现 `submitWordProgress`（带幂等）
4. 实现 `finishSession` 与会话完成判定
5. 接入前端 `index/card/learn/spell`
6. 联调 `getTodayPlan` 与首页统计

---

## 11. 验收清单（最小）

- 用户进入学习模式，首次创建 20 词以内会话成功
- 学到一半退出，再进页面返回同一 `session_id` 与同一组词
- 每次对/错提交后刷新页面，进度不丢失
- 重复点击导致重复请求时，进度不重复累计
- 会话完成后状态变更为 `finished`，下次进入新会话

---

## 12. 结论

你提出的整体流程是可行的。  
要稳定落地，关键是把“页面阶段”抽象为后端可理解的“会话 + 单词进度 + 幂等提交”。  
按本文设计实现后，可以同时满足：

- 一组最多 20 词
- 中途退出可恢复
- 每次操作实时保存
- 学习与复习模式统一治理
